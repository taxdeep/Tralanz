## Deferred — explicit followups

### Phase 3.1 — Hydrate the 5 P2 Create pages

Each Create page needs:

1. `[SupplyParameterFromQuery(Name = "copyFrom")] public Guid? CopyFromId { get; set; }`
2. `_isFromCopy` bool field + "This is a copy" `CitusAlert`
3. `TryHydrateFromCopySourceAsync` method called from `OnInitializedAsync`
4. Per-entity hydration heuristics (see Phase 1 / 2 PRs for the pattern):
   - Customer / vendor → copy
   - Document date → today; due date → today + 30
   - Currency → from source
   - FX rate → null (let FxRateInline re-resolve)
   - Memo → copy
   - Lines → copy (sans TaxCodeId where read DTO doesn't expose it)
   - Task back-link → don't copy (source task may be done)
   - Customer PO # / Cheque # / Ref # → blank (per-occurrence)

### Batch C — Sub-account hierarchy

Drafted then reverted (branch `coa-batch-c-d-subaccount-lock` —
deleted locally, never pushed). Re-entry plan:

1. **Migration**: `deploy/migrations/2026-05-25-coa-subaccount-and-lock.sql`
   - `accounts.parent_account_id uuid null references accounts(id) on delete set null`
   - Partial index `where parent_account_id is not null`
   - No CHECK forcing same `root_type` parent/child (rare but valid)
2. **Domain record**: add `ParentAccountId` to `Modules.Company...
   ChartOfAccounts.AccountRecord` (or wherever the canonical record
   lives).
3. **HTTP DTO**: add `ParentAccountId` to `AccountUpsertHttpRequest`
   in the Accounting.Api.
4. **Store**: `PostgreSqlAccountStore` insert + update SQL must
   include the new column.
5. **Client DTO**: `AccountSummary` + `AccountUpsertPayload`
   gain `ParentAccountId`.
6. **UI**:
   - `ChartOfAccountsPage` drawer: "Make this a subaccount" checkbox
     + parent picker that filters to active siblings of the chosen
     root type, excluding self when editing.
   - Main list: indent rows whose `ParentAccountId` is set; sort
     children directly after their parent.
   - Preview pane (already shipped in Batch A): consume the parent
     id to render the new row indented under its parent if any.

### Batch D — Lock Account toggle

Same migration file as Batch C (drafted together; both reverted).
Re-entry plan:

1. **Migration** (same file as Batch C):
   - `accounts.locked_at timestamptz null`
   - `accounts.locked_by_user_id char(7) null`
   - Partial index `where locked_at is not null`
2. **Store enforcement**: in `PostgreSqlAccountStore.UpdateAsync`,
   read `locked_at` first; if non-null AND the patch touches any of
   `code / name / root_type / detail_type / currency_code`, raise
   `InvalidOperationException("locked")`. Allow toggling `is_active`
   and `locked_at` itself.
3. **Audit**: write `account_locked` / `account_unlocked` to
   `audit_logs` on each state flip. Payload includes the actor and
   prior value.
4. **UI**:
   - `ChartOfAccountsPage` drawer: existing "Lock account" text +
     icon already exists; wire it to a state toggle bound to the
     new column.
   - Main list: render a lock icon on locked rows.
   - On Edit attempt for a locked account, show a banner "Unlock
     this account to edit name / code / type / currency".
